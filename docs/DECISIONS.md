## 2026-06-09 — R&D 코드를 복사하지 않는다

### 결정
Project Signal Personal은 Project Signal R&D 코드를 복사하지 않고 처음부터 새로 만든다.

### 이유
R&D는 실험 기록이고, 본게임은 안정 공간이어야 한다. 실험 코드의 구조적 부채를 본게임에 이식하지 않는다.

### 제외한 것
R&D 코드 일부를 베이스로 사용하는 방안.

### 이후 영향
Phase 0부터 모든 코드를 직접 작성. 빌드 도구 없이 순수 HTML/CSS/JS로 진행.

## 2026-06-09 — 성장 구조는 파티 강화로 시작한다

### 결정
Phase 7 성장 선택지는 직업 강화가 아니라 파티 전체 강화(공격 +1 / 최대 HP +5)로 시작한다.

### 이유
초반에는 파티 전체가 함께 강해지는 감각이 더 중요하다. 직업별 강화는 파티 구성이 다양해진 이후에 의미가 생긴다.

### 제외한 것
직업별 고유 스킬 강화, 개인 스탯 강화.

### 이후 영향
run.bonuses { atk, maxHp }로 누적. 다음 스테이지 파티 생성 시 반영. Phase 7.8에서 전투 화면에 현황 표시 추가.

## 2026-06-09 — Phase 8.1은 표시 구조만, slotIndex는 미추가

### 결정
Phase 8.1에서는 unit.slotIndex를 state/unit에 추가하지 않는다. render 레이어에서 배열 인덱스로 슬롯을 매핑한다.

### 이유
아직 배치 선택 시스템이 없는 단계에서 slotIndex를 추가하면 의미 없는 구조가 state에 들어간다. Phase 9 배치 선택 UI 시점에 확정하는 것이 맞다.

### 제외한 것
unit.slotIndex를 state에 미리 추가하는 방안.

### 이후 영향
Phase 8.1은 render.js / styles.css만 변경. battle.js 무변경. Phase 9에서 slotIndex 도입 시 state.js 수정 예정.

---

## 2026-06-09 — Phase 8.2 대각 전장은 루다 polish 확인 후 결정

### 결정
세로형 대각 전장 레이아웃은 루다 vertical battlefield polish 결과를 확인한 뒤 본게임 이식 범위를 결정한다.

### 이유
대각 레이아웃은 CSS 구조 변경 폭이 크다. 루다에서 시안이 확정되기 전에 본게임에서 먼저 잡으면 나중에 두 번 엎게 된다.

### 제외한 것
Phase 8.1과 8.2를 한 번에 진행하는 방안.

### 이후 영향
Phase 8.2는 루다 presentation-lab Battlefield Layout Prototype polish 완료 후 진행.
