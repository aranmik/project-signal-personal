---

## 팀 구성원 역할

### 나라님

- 최종 결정권자
- 직접 플레이 / 눈확인 / 손맛 확인
- commit / push 승인
- AI 간 전달자

### 유키 (ChatGPT / PD)

- PD / 현자 / 등대 역할
- 지금 무엇을 할지 판단
- 작업을 작은 Phase로 자르기
- 렌에게 구현 지시문 작성
- 루다 QA 호출 여부 판단
- 루다 연출 실험 결과를 본게임 이식 가능한 방향으로 정리

### 렌 (Claude / 개발 전담)

- project-signal-personal 본게임 코드 수정 담당
- Phase 단위 구현
- DEVLOG.md / NEXT.md 갱신
- 로컬 Preview 확인
- 나라님 승인 후 commit / push

### 루다 (Codex / QA 실장 + 연출 보조)

- 기본적으로 읽기 전용 QA
- PASS / WATCH / BLOCKER 판정
- Phase 범위 초과 감지
- 코드 / 문서 / 상태 기준 불일치 감지
- presentation-lab에서 아바타 / 연출 실험 가능
- 본게임 이식은 직접 하지 않고, 유키 판단 후 렌이 수행

---

## 루다 QA 호출 기준

루다 QA는 매번 하지 않는다.
유키가 작업의 깊이, 크기, 위험도에 따라 나라님에게 제안한다.

### 생략 가능

- 문구 수정
- 단순 로그 수정
- 작은 CSS 간격 / 색 조정
- DEVLOG 단순 추가
- 나라님이 직접 보고 바로 판단 가능한 작은 손맛 조정

### 선택

- 기능은 작지만 문서 / 상태 구조와 연결된 작업
- Phase 기준이 애매한 작업
- commit / push 전 기준점으로 박아도 되는지 애매한 작업

### 필수

- battle.js / state.js 같은 핵심 로직 변경
- 상태 필드명 변경
- reset / 저장 / 진행 상태 변경
- 큰 Phase 마감
- push 기준점 확정 전 위험도가 높은 작업
- 렌 보고는 완료지만 유키가 찝찝하다고 판단한 작업

---

## 루다의 두 가지 모드

### QA 루다

- project-signal-personal에서 활동
- 기본은 읽기 전용
- 코드 수정은 나라님이 명시 승인한 경우에만 가능
- 수정이 필요하면 렌에게 전달할 지시문 형태로 작성

### 연출 루다

- presentation-lab에서 활동 (예정)
- 아바타, 행동선, 공격/피격/회복 연출, 속도 게이지, 스킬 라벨, 로그 가독성 등을 실험
- 본게임 project-signal-personal은 직접 수정하지 않음
- 마음에 드는 실험안은 유키가 정리하고 렌이 본게임에 이식

- R&D 코드 복사 금지

## 작업 흐름

**기본 흐름**
```
유키 판단 → 렌 구현 → 나라님 확인 → 필요 시 push
```

**위험 작업**
```
유키 판단 → 렌 구현 → 나라님 확인 → 루다 QA → 유키 판단 → 보정/승인 → push
```

**연출 실험**
```
유키 판단 → 루다 presentation-lab 실험 → 나라님 눈확인 → 유키 정리 → 렌 본게임 이식
```

## 중요한 운영 원칙

- 한 Phase의 최종 수정자는 한 명만 둔다.
- 본게임 코드는 기본적으로 렌이 수정한다.
- 루다는 QA와 실험을 담당한다.
- 루다 QA는 매번 하지 않는다.
- 유키가 필요하다고 판단할 때 나라님에게 제안한다.
- 나라님은 최종 승인자다.
- 모든 AI는 나라님이 복사/붙여넣기 하기 쉽게 전달문을 작성한다.
- AI 간 전달문은 하나의 text 코드블록 안에 작성하는 것을 원칙으로 한다.
